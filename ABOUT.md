# About heymark

heymark는 AI 기반 개발 워크플로우를 위한 마크다운 중심의 개인 블로그 시스템입니다. Cursor나 Claude Code 같은 AI 코딩 도구와의 통합을 최우선으로 설계되었으며, 개발자가 문서를 빠르게 작성하고 공유할 수 있는 환경을 제공합니다.

heymark는 단순한 콘텐츠 공유 도구를 넘어, **AI 개발을 위한 skill 아카이브**이자 **문서 기반 협업 플랫폼**으로 설계되었습니다. 개발 과정에서 발견한 개념, 패턴, 규칙을 지속적으로 기록하고 정리하며, 이를 Claude Skill의 reference 문서로 활용할 수 있는 통합 시스템입니다.

## 프로젝트 개요

### 기존 플랫폼의 한계

기존 블로그 플랫폼(티스토리, Velog, Notion 등)은 AI 기반 개발 워크플로우에 최적화되어 있지 않습니다.

**접근성과 공유의 문제**

- 마크다운 문서를 AI에게 전달하려면 복사-붙여넣기 과정이 필요합니다
- 공개용과 프라이빗 문서를 구분하려면 별도의 플랫폼을 사용해야 합니다
- Notion의 게시 기능은 URL이 `notion.so/workspace/page-title-32자리-랜덤-문자열`처럼 복잡하고, 특정 사람에게만 액세스 권한을 주려면 워크스페이스 초대나 복잡한 설정이 필요합니다

**편집과 커스터마이징의 제약**

- Velog는 디자인 수정이 불가능합니다
- 티스토리는 마크다운 작성이 불편합니다
- 이미지나 동영상을 원하는 크기와 위치에 배치하는 것도 제한적입니다

**개발 워크플로우와의 단절**

- 개발 환경에서 문서를 작성한 후 별도의 플랫폼에 복사-붙여넣기 해야 합니다
- 문서 버전 관리가 어렵고, Git과의 통합이 제한적입니다
- CLI 기반의 빠른 배포가 불가능합니다

### AI 개발 워크플로우의 본질

바이브 코딩(Vibe Coding)의 핵심은 **마크다운으로 계획 → AI가 구현 → 개발자 검토 및 피드백**의 반복입니다. 이는 "대화로 구현하는 것"이 아니라 **"대화로 일을 운영하는 것"**입니다. 요리사(개발자)가 레시피(의도)를 작성하고, 조리사(AI)가 요리를 만들고, 요리사가 맛을 확인하는 것과 같습니다.

이 과정에서 마크다운 문서는 적은 토큰으로 높은 품질의 결과를 얻는 핵심 자산입니다. 특히 Claude Skill과 같은 시스템에서는 문서의 frontmatter를 기반으로 적절한 skill을 자동으로 선택하여 적용하므로, 잘 구조화된 문서 아카이브가 필수적입니다.

### Claude Skill의 작동 원리

Claude Skill은 등록된 Markdown 파일들의 **frontmatter 메타데이터**를 먼저 스캔한 뒤, 사용자의 요청과 가장 부합하는 문서를 선택하여 해당 내용을 내부 컨텍스트로 로드하고 응답을 생성합니다.

각 reference 문서는 반드시 frontmatter를 포함해야 합니다. Frontmatter는 **책의 목차와 색인** 역할을 합니다. Claude는 frontmatter만 보고 적절한 문서를 선택합니다.

```yaml
---
title: Viewer Camera Control Pattern
description: Autodesk LMV 환경에서 카메라 이동과 애니메이션을 제어하는 공식 패턴 정리
tags: [autodesk, lmv, camera, animation]
---
```

Claude는 다음 순서로 동작합니다:

1. 모든 문서의 **frontmatter만 빠르게 훑습니다**
2. 사용자의 요청 의도와 가장 잘 맞는 문서를 **선별**합니다
3. 선택된 문서의 **본문을 실제로 읽습니다**
4. 해당 문서의 내용을 기준으로 코드/설명을 생성합니다

이 구조의 장점:

- 문서가 수천 개여도 frontmatter만 보고 빠르게 찾을 수 있습니다
- 여러 문서를 섞어서 엉뚱한 답을 주는 문제를 줄일 수 있습니다
- 사용자가 원하는 정확한 문서를 찾을 수 있습니다

바이브 코딩에서 Claude Skill은 **"레시피 책"**과 같습니다. 요리사(개발자)가 매번 레시피를 새로 설명하는 대신, 레시피 책에서 적절한 레시피를 찾아서 조리사(AI)에게 주는 것과 같습니다.

Claude Skill의 본질은:

> **"Claude가 나 대신 일관된 판단을 하도록 만드는 기준 세트"**

입니다. 즉, Skill은 결과물이 아니라 **나만의 커닝페이퍼**입니다. 시험 볼 때 필요한 공식만 적어둔 종이처럼, Claude가 필요할 때만 꺼내서 쓰는 나만의 기준서입니다.

### Skill 구축의 어려움

Claude Skill의 개념을 이해하고 몇 개의 문서를 작성하는 것은 어렵지 않습니다. 진짜 어려운 것은:

- **지속성**: 새로운 패턴을 발견할 때마다 문서화하는 습관을 유지하는 것
- **접근성**: 어디서나 문서에 접근하고, 필요할 때 빠르게 찾고 수정할 수 있는 환경
- **자동화**: 문서를 작성하고 배포하는 과정이 일상 업무에 자연스럽게 통합되는 것

개발자는 매일 새로운 패턴을 발견하고, 기존 개념을 개선합니다. 하지만 이 과정을 수동으로 관리하려면:

- 문서 작성에 별도의 시간을 할애해야 하고
- 파일 시스템에서 문서를 찾고 관리하는 부담이 생기고
- 배포하고 업데이트하는 것이 번거로워집니다

결국 문서화를 시작했지만, 지속적으로 문서를 추가하지 못하게 됩니다. 문서가 적으면, AI는 제대로 된 답을 줄 수 없습니다.

### heymark의 해결책

heymark는 **개발과 문서화의 경계를 없애는 것**을 목표로 합니다. 개발자가 코드를 작성하는 과정에서 자연스럽게 개념을 정리하고, AI의 도움을 받아 문서의 품질을 높이며, 즉시 Skill에 반영할 수 있는 환경을 제공합니다.

다음과 같은 워크플로우를 통해 개발자는 별도의 문서 작성 시간을 할애하지 않고도, 자연스럽게 Claude Skill의 reference 문서를 축적할 수 있습니다:

1. **즉시 기록**: 개발 중 발견한 개념이나 패턴을 즉시 마크다운으로 기록합니다
2. **AI 협업**: Claude나 Cursor 같은 AI 도구와 협업하여 문서의 구조와 내용을 다듬습니다
3. **자동 저장**: CLI 명령어 하나로 문서를 구조화된 저장소에 추가하고 배포합니다
4. **지속적 개선**: 필요에 따라 문서를 수정하고 업데이트하여 Skill의 품질을 지속적으로 향상시킵니다

체계적인 문서 관리 시스템이 갖춰져 있을 때, Claude Skill은 정적인 문서 모음이 아니라 **개발 과정과 함께 성장하는 살아있는 지식 생태계**가 됩니다.

## 핵심 기능 (구현 완료)

### CLI 기반 빠른 문서 관리

터미널에서 몇 번의 명령으로 문서를 작성하고 배포할 수 있습니다. heymark CLI의 `call` 명령어로 AI에게 전달할 프롬프트(어떤 형식 및 말투로 md 문서를 만들어줘~)를 클립보드에 복사하고, `add` 명령어로 작성한 마크다운 파일을 heymark에 즉시 추가합니다. 별도의 웹 인터페이스나 복잡한 설정 없이 개발 환경에서 바로 작업할 수 있습니다.

이를 통해 개발 과정에서 발견한 개념을 즉시 마크다운으로 정리하고, CLI 명령어 하나로 Skill에 반영할 수 있는 환경을 제공합니다.

### 유연한 접근 권한 관리

## 기술적 특징

### 코드와 콘텐츠의 분리

블로그 애플리케이션 코드는 Public Repository에, 실제 마크다운 포스트는 Private Repository에 저장됩니다. 코드는 오픈소스로 공개하면서도 콘텐츠는 안전하게 관리할 수 있습니다. GitHub API를 통해 private repository의 포스트를 동적으로 가져오며, frontmatter 기반의 권한 관리로 접근을 제어합니다.

이러한 구조는 Git 기반 버전 관리를 제공하여 문서의 변경 이력을 추적하고 롤백할 수 있게 하며, 지식의 진화 과정을 관리할 수 있습니다.

### 확장 가능한 아키텍처

React + Vite 기반의 프론트엔드와 Vercel Serverless Functions 기반의 백엔드로 구성되어 있습니다. 컴포넌트와 API 라우트가 명확히 분리되어 있어, 새로운 인증 방식을 추가하거나 마크다운 렌더링 로직을 변경할 때 다른 부분에 영향을 주지 않습니다.

Vercel의 서버리스 함수는 자동 스케일링과 글로벌 CDN을 제공하므로 트래픽이 증가해도 별도의 인프라 관리 없이 안정적으로 운영할 수 있습니다.

## 바이브 코딩에서의 실질적 가치

### 토큰 예산 관리

바이브 코딩의 가장 현실적인 장벽은 대화가 길어지면서 품질이 흔들리는 것입니다. 토큰은 무제한이 아니라 예산입니다. 긴 설명을 매번 반복하는 것은 비효율적이며, 토큰을 가장 많이 잡아먹는 것은 '중복 설명'입니다.

heymark의 frontmatter 기반 문서 관리를 통해 Claude는 전체 문서를 읽지 않고 **frontmatter만 빠르게 훑어서** 적절한 문서를 선택합니다. 이는 바이브 코딩에서 강조하는 "전문가처럼 근거 인용만 추출한다"는 원칙과 정확히 일치합니다.

### 워크플로우의 고정

바이브 코딩을 잘 활용하는 개발자는 "매번 레시피를 새로 설명하는 사람"이 아니라, **검증된 레시피를 레시피 책에 정리해두고 재사용하는 사람**입니다. heymark는 바로 그 레시피 책입니다.

각 문서는 하나의 검증된 레시피가 되어, 매번 긴 설명 없이도 Claude가 일관된 결과를 만들 수 있게 합니다.

### 사고 방식의 외주화

바이브 코딩의 본질은 "시키는 일"에서 **"목표 기반 역할 설계"**로의 전환입니다. 개발자가 맡아야 하는 핵심은 의도 작성, 계획 수립, 결과 검증입니다. 나머지는 AI에게 위임합니다.

heymark는 이 위임의 핵심 도구입니다. 개발자가 "이 패턴은 이렇게 해야 한다"는 기준을 문서에 적어두면, AI는 그 문서를 자동으로 참조하여 일관된 결과를 만듭니다.

### 의도 문서화의 자동화

바이브 코딩에서 의도(요구사항)를 문서로 남기는 것은 필수입니다. 하지만 매번 문서를 새로 작성하는 것은 비효율적입니다. 대신, 개발하면서 발견한 좋은 방법을 즉시 문서화하고, AI와 협업하여 다듬고, 자동으로 배포할 수 있어야 합니다.

heymark는 CLI 통합을 통해 개발 환경에서 빠르게 문서를 작성하고 배포할 수 있게 하며, 웹 접근성을 통해 브라우저를 통해 어디서나 문서에 접근하고 공유할 수 있게 합니다.

## 향후 계획

### 오픈소스 생태계 구축

heymark를 오픈소스로 운영하는 이유는 다양한 개발자들의 피드백과 기여를 통해 지속적으로 개선될 수 있고, 실제 사용자들의 다양한 니즈를 반영할 수 있기 때문입니다. 또한 heymark의 아키텍처와 구현 방식을 공개함으로써, 다른 개발자들이 AI 기반 개발 워크플로우를 구축하는 방법을 학습하고 자신만의 변형을 만들 수 있어 전체 개발 커뮤니티의 지식 공유에 기여합니다.

다른 개발자들이 heymark를 fork하여 자신만의 블로그를 만들 수 있도록 상세한 설정 가이드와 환경 변수 템플릿을 제공합니다. `.env` 파일 설정, GitHub Personal Access Token 발급, Google OAuth 설정 등 초기 설정 과정을 단계별로 문서화하여 30분 이내에 배포 가능하도록 합니다.

### 미디어 업로드 기능

Cloudflare 기반으로 이미지 또는 동영상을 쉽게 업로드하고 URL을 부여하여 md에 이미지와 동영상을 더 쉽게 추가하고 원하는 크기와 위치에 배치할 수 있는 기능을 개발합니다.

### API 성능 최적화

현재는 게시물을 불러오기 위해 GitHub API를 사용하는데, 시간당 호출 제한이 있어 캐싱 전략을 마련해야 합니다.

### 콘텐츠 조직화

폴더 구조를 기반으로 시리즈나 카테고리를 자동으로 인식하여 게시물을 묶는 기능을 추가합니다. 예를 들어 private repository에 `react-series/part1.md`, `react-series/part2.md`처럼 폴더로 정리하면, 웹에서 자동으로 "React 시리즈"라는 그룹으로 묶어서 표시합니다. 별도의 카테고리 설정이나 태그 관리 없이 파일 시스템 구조만으로 콘텐츠를 조직화할 수 있습니다.

### 지속 가능한 운영

오픈소스 생태계를 유지하면서도 핵심 기능을 지속적으로 개선할 수 있는 수익화 방안을 검토합니다.

## 결론

heymark는 단순한 콘텐츠 공유 도구를 넘어, AI 개발 워크플로우의 핵심 인프라입니다. Frontmatter 자동 생성, Git 기반 버전 관리, CLI 통합, 웹 접근성, 권한 관리 등을 갖춘 heymark를 통해 개발자는 별도의 노력 없이도 자연스럽게 Claude Skill을 축적하고 개선할 수 있습니다.

지금 정리하는 문서들은 나중에 AI가 **"필요할 때만 꺼내 쓰는 커닝페이퍼"**가 될 것입니다. 도서관에 책이 많을수록, 사서는 더 정확한 답을 줄 수 있습니다. 개발과 문서화의 경계를 없애고, 개발 과정에서 자연스럽게 개념을 정리하고 Skill에 반영할 수 있는 환경이 있을 때, Claude Skill은 정적인 문서 모음이 아니라 **개발 과정과 함께 성장하는 살아있는 지식 생태계**가 됩니다.
